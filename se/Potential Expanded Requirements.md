# Potential Expanded Requirements

Requirment archive for the defunct `iotserver` project.

## Stakeholder and System Requirements
A code starting with CS means that the requirement is derived from cybersecurity needs and desires, BS indicates a basic requirement.

* CS1: The system must reject any requests not sent by an approved node
* CS2: The architecture should apply key network security principles including segmentation, isolation, centralized logging, network monitoring, and malicious code protection and incorporate Zero Trust Architecture (ZTA) considerations tailored to an OT environment.
* CS3: The architecture should employ firewalls as boundary protection devices to enforce network isolation and control connections and information flows between segments.
* CS4: The architecture should maintain and track device trust in accordance with the system model and policy, leveraging embedded hardware security technologies (e.g., TPM, AES, SHA).
* CS5: The architecture should include comprehensive software and endpoint security controls, including monitoring and analysis, secure configuration and management, endpoint hardening, integrity protection, access control, device identity, root of trust, and physical security.
* CS6: The architecture should incorporate software security protection mechanisms—application allowlisting, regular patching, secure code development, and configuration management including application hardening—to ensure OT applications and services are used and maintained properly.
* BS1: The system must be replicable as a infrastructure-as-code template
* BS2: The system nodes must be able to interface with both sensors and actuators
* BS3: The nodes must be able to compute simple metrics (i.e. averages, min, max) on-device over a small window of locally stored measurements
* BS4: The nodes must be able to log measurements and simple metrics to a centralized database at a remotely configurable frequency
* BS5: Each on-device metric must be implemented as a reusable class
* BS6: The nodes must implement a simple on-device rules engine (i.e. based on simple metrics and thresholds) for on-device anomaly detection or actuator control, and all events generated by the engine must be logged to a centralized database
* BS7: The on-device rules engine parameters must be remotely configurable
* BS8: The system should not require the nodes to have greater processing power than an ESP32-S3 for on-device calculations
* BS9: The system users must be able to configure complex data processing workflows (i.e. for custom metrics or trend analytics) on database data using centralized rules engines
* BS10: The system users should be able to add and remove edge data processors, which act as rules engines at the edge for metrics, anomaly detection or actuator control with intermediate processing power
* BS11: The system users should be able to train and deploy cloud AI models for data analysis (no control)
* BS12: The system users should be able to train and deploy cloud AI models for high-latency control, and their control events must be logged to the centralized database
* BS12: The system users should be able to train and deploy edge AI models for real-time control, and their control events must be logged to the centralized database
* BS13: The system users must be able to access and configure dashboards which show and filter measurements, events generated by on-device rules engines, node statuses, events generated by edge and centralized rules engines, AI analyses, and AI control events
* BS14: The system users must be able to access and configure control panels to manually send commands to the sensors interfacing with actuators
* BS15: The system users must be able to deploy OTA updates to nodes and edge data processors
* BS16: The system users must be able to share data with external parties
* BS17: The system should avoid vendor-locked solutions and allow migration between cloud services, server providers, and on-premises deployments.
* BS18: The system should not require node owners to have specialized IT teams and must be manageable by existing non-technical personnel.
* BS19: The system should minimize operational and deployment costs
* BS20: The system must quantify node processing load based on the number of interfaced sensors and actuators, metrics computed, rules evaluated, and logging and control frequencies, and alert users if the load is too high
